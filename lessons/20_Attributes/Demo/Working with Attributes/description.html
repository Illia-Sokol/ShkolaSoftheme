Introduction

This sample demonstrates the creation and use of custom attributes to extend and modify the behavior of code. The .NET Framework uses attributes extensively to provide automatic generation of WSDL contracts for Web Services, Platform Invoke (P/Invoke) services, and design-time support for custom controls, among several other uses. Developers can define their own attributes and look for them when examining arbitrary code (for example, creating a designer or providing support for third-party plug-ins).

Contents

The self-extracting archive contains all source code for this sample application. The file contains the following directories:

  • Code: Contains the Visual Studio .NET solution file.
    • AttributeUser: Contains the source code for the Windows Forms client project that uses the custom attribute.
    • TraceLibrary: Contains the source code for the Class Library project that defines the custom attribute and its behavior.
  • Misc: Contains any additional files required to run the sample application.

How to Use this Sample Application

In order to run this sample application, run the self-extracting archive and then double-click the AttributeSample.sln file to open the solution in the Visual Studio .NET development environment.

Click the Start button on the Toolbar or press the F5 key to run the application. In the main form, a list box will show a number of predefined “items” and a text box is provided so that new items may be added. The content of these items is irrelevant, since they are simply treated as strings by the application. When the “Process Items” button is clicked, each item in the list box is processed by the ItemProcessor class, which is tagged with the custom TraceAttribute. Upon completion, a dialog box is displayed asking the user if they wish to view the log file.

Developers may use the application configuration file, included with the AttributeSample project, to modify the amount of information that is logged and where the information is recorded.

Description

The ItemProcessor class is generically named because it represents nearly any functionality a developer may wish to provide. The core functionality of the class bears little relation to the value-added functionality provided by usage of the TraceAttribute. In this sample, developers may wish to extend the ItemProcessor class with other methods and operations to verify that the tracing behavior is independent of the operation of the class.

An attribute provides developers with the means to “tag” their code with information that enables specific behavior when the code is used in certain contexts. For example, by inheriting the ItemProcessor class from ContextBoundObject and tagging that class with the TraceAttribute (which inherits from ContextAttribute), the ContextBoundObject ensures that any ContextAttribute objects that are present will be “activated”. It is through this activation that the tracing behavior is provided to instances of that class.

Attributes alone are insufficient to provide new behavior or functionality and the runtime can safely ignore custom attributes at runtime, but when combined with classes that recognize specific attributes and know what services to provide for them, they become a powerful addition to any developer’s toolbox.
